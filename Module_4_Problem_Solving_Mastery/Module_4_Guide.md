# MODULE 4 — Problem-Solving Mastery + Capstone (Days 25-30)

**Goal:** Stop learning new syntax and start *training the brain*. This week is daily problem-solving reps using everything you know, plus a capstone program you build entirely from scratch — no AI. This is where "I go blank" finally dies.

**How every day works:** pick the day's problems, and for EACH one strictly follow UNDERSTAND → PLAN (on paper) → CODE → TEST → DEBUG (see `CHEATSHEET.md`). Time-box: if stuck 15 min, use the stuck-routine, not AI.

---

## Day 25 — Problem-Solving Reps I
### DO (blank files, no AI)
- Sum of digits of a number (e.g. 1234 → 10).
- Reverse a number (1234 → 4321) without converting to a string if you can.
- Count how many words are in a sentence.
### TRANSITION CONDITION
You solved all three using the written framework, with your plan written before code.

---

## Day 26 — Problem-Solving Reps II
### DO
- FizzBuzz again — but from total memory, fast.
- Find the largest word in a sentence.
- Check if two words are anagrams (same letters rearranged). Plan carefully on paper.
### TRANSITION CONDITION
You solved the anagram problem yourself, explaining your approach out loud.

---

## Day 27 — Debugging Bootcamp
### LEARN
Real coding is mostly fixing code that doesn't work yet. Skill: read the error, isolate the line, `console.log` your variables, fix, re-test.
### DO
- Take any 2 solutions you wrote earlier and INTENTIONALLY break them (typo, wrong condition, off-by-one). Then debug them as if you didn't know the cause.
- Practice reading 3 different error messages and explaining what each means.
### TRANSITION CONDITION
You can take broken code, read the error, find the bug, and fix it — calmly, no AI.

---

## Day 28 — Capstone Planning
### LEARN
Pick ONE program to build from scratch. Suggestions (choose one):
- **Number guessing game** (computer picks a number, you guess, it says higher/lower, counts attempts).
- **To-do list in the terminal** (add/list/remove tasks held in an array).
- **Simple quiz** (asks questions, tracks score, reports result).
### DO
- Write your plan ON PAPER: what data you store, what functions you need, the step-by-step flow. Decompose it into small functions.
- Create the project folder + file. Stub out the function names with comments.
### TRANSITION CONDITION
You have a written plan and a decomposed list of functions before writing real logic.

---

## Day 29 — Capstone Build
### DO (no AI — this is the proof)
- Build your capstone function by function. Test each piece as you go.
- Keep a stuck-log. Solve problems with the framework, docs, and your brain only.
### TRANSITION CONDITION
Your capstone's core features work end-to-end, written entirely by you.

---

## Day 30 — Capstone Finish + Graduation 🎓
### DO
- Finish, clean up, and add comments explaining your logic in your own words.
- Write a short `README` for your capstone: what it does, how to run it, what you learned.
- **Then do this:** open a blank file and re-solve FizzBuzz and "reverse a string" with a timer. Notice you don't freeze anymore.
### FINAL TRANSITION CONDITION (graduation)
1. You built a working program from scratch with zero AI.
2. You can sit at a blank file and solve a small problem using the framework, calmly.
3. You can read an error and debug yourself.
4. You can explain your own code line by line.
- **If yes: the "I go blank" feeling is gone. You can code. Now go make No-Plan-B easier than it would have been.**
